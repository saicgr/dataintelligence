Single source-of-truth backlog of technical terms pulled from job descriptions, cross-referenced
against the mobile app's content (`mobile/src/lib/generated/*.json` + the track registry in
`mobile/src/lib/content.ts`). The **jd-term-extractor** agent appends/updates this file — it is the
ONLY file that agent writes, and it upserts (dedupes by term) on every run. The **content-ingestor**
agent reads this file to decide what to author next.

**Coverage target: every track should have ≥ 25 cards, with a healthy spread across all five levels
(Jr · Mid · Sr · Staff · Principal).** The **Track Coverage** table is the supply/health view (are we at
25? which levels are empty?); the **Terms** table is the demand backlog (what JDs ask for, what's missing).

**Term Status legend**
- `✅ covered` — the term has ≥10 dedicated mentions in its mapped track AND the track has ≥25 cards
- `🟡 thin` — 1–9 dedicated mentions in its mapped track, or the track is under 25 cards; includes sub-topics that may warrant their own track at high demand
- `🔵 untracked` — cards mention it across tracks but no track name matches (lives inside another track, no dedicated home)
- `⬜ missing` — zero cards mention it anywhere → needs authoring (counts blank)
- *(blank)* — not yet looked up

`# Q&A` = cards in the mapped track that mention the term (blank if none).
`Track` = mobile track name, blank if track is missing; a trailing `?` (e.g. `aws?`) marks a weak/ambiguous match to review.
`# JDs` = demand — how many distinct JDs requested this term. `Suggested Action` = the next step for the
content-ingestor (`author N` / `create track` / `skip — covered` / `review`). `Date Added` = date first
seen (YYYY-MM-DD, never overwritten). `Seen in JD` = the JD label(s) it appeared in. Leave any unknown cell
blank — never guess.

**No duplicates.** One row per canonical term. The agent normalizes via the **Alias registry** below
(`Apache Airflow` → `Airflow`) before comparing, so a term never appears twice — and a term already
`✅ covered` in the mobile app is flagged `skip — covered` so the content-ingestor doesn't re-author it.

**Sort:** highest priority first = needs-work (missing → untracked → thin) before covered/blank, then by
`# JDs` (demand) descending, then alphabetically. So the most-requested gaps float to the top.

**Sub-topic grading rule (applied from 2026-06-08):** a sub-topic that lives inside a parent track (e.g.
Redshift in aws, Kinesis in aws) is graded on its OWN dedicated mention count: ≥10 mentions = `✅ covered`,
1–9 = `🟡 thin · consider own track` (if high demand), 0 = `⬜ missing`. The parent track's card total does
NOT automatically confer "covered" status on a sub-topic.

## Track Coverage — are we at 25, and where are the level gaps?

Per-level counts come from each card's `level` field in `mobile/src/lib/generated/<slug>.json`
(`Jr`/`Mid`/`Sr` are tagged today; `Staff`/`Principal` show 0 until authored — that 0 IS the gap).
`Status`: `⬜ +N to 25` if under target · `🟡 25+ · gaps: <empty levels>` if at 25 but a level is empty
· `✅ 25+ balanced` if at 25 with all five levels populated. The agent refreshes a track's row whenever
this JD touches a term that maps to it.
